# FDD 系统开发文档 v1.0

报告时间：2026-07-02

报告来源：Claude Fable5



配套《FDD 项目方案与待确认清单 v1.1》。方案文档回答"做什么、为什么";本文档回答"怎么造、验收标准是什么、按什么顺序"。

## 0. 文档范围声明

本文档在以下意义上完整:架构、技术栈、数据契约、模块规格、验收标准、里程碑全部给出,工程师可直接按 M0 开工。两处接口留桩(非设计缺口,是外部交付未到):

- **C3 平台故障事件契约**:按 v1.1 附录 B 规格写桩,平台实际 schema 到达后仅调整字段映射层,不动引擎。
- **遥测交付机制**:D-1 审批决定数据以何种形态到达(批量导出文件/库直连/API)。本文档按"批量 Parquet/CSV 切片交付"设计摄取层,其他形态只改摄取适配器。

## 1. 系统架构

```
[平台DB(生产,不动)] --D-1审批切片--> [摄取适配器] --> [分析数据湖 Parquet]
[Zoho 标注流水线 v2] ----------------------------------^
[实验室数据] [产线数据] ---------------------------------^
                                    |
                          [DuckDB 查询层]
                                    |
   M-CONV 物理转换 --> M-SEG 分段/稳态 --> M-SENSE 传感器信任
                                    |
                          M-FEAT 特征库(同工况分箱)
                                    |
        M-BASE 三层基线  <-->  M-DRIFT 漂移检测(CUSUM)
                                    |
                    M-DIAG 诊断组合(树模型+症状矩阵)
                                    |
        M-LABEL 标签连接 --> M-VALID 验证机架 --> 诊断输出五件套
                                    |
                    [聚合出域层: 只出统计与模型,零PII]
```

全流程本地运行,无外部服务依赖。这是安全审批的承诺,也是架构约束:任何模块引入外部 API 调用即违约。

## 2. 技术栈决定(声明制,附理由与可逆性)

| 决定 | 理由 | 可逆性 |
|---|---|---|
| Python 3.11+ | 团队现有脚本生态(Zoho 流水线已是 Python) | 不可逆也无需逆 |
| 存储:Parquet 文件湖,按 `sn_hash/year/month` 分区 | 10TB 级列式压缩后约 1–2TB;免运维;符合"分析副本不动生产库"原则 | 若后续需多人并发在线查询,可平迁 ClickHouse,分区设计兼容 |
| 查询:DuckDB | 单机扫 TB 级 Parquet 足够;零部署 | 同上 |
| 物性:CoolProp(R410A) | 消灭查表近似;开源本地 | REFPROP 是付费升级项,精度差异对本项目无感 |
| 建模:scikit-learn + XGBoost | 方法上限已锁定树模型;可解释性工具链成熟(SHAP) | — |
| 漂移:自实现 CUSUM/鲁棒 Z(约 200 行) | 无成熟依赖值得引入 | — |
| 环境:公司内隔离分析机,依赖锁定 requirements.txt | 安全架构要求 | — |

## 3. 数据契约

**C1 遥测契约(已锁定,48 列)**。转换规则表——摄取时即物化以下派生列,原始列保留:

| 派生列 | 规则 | 依据 |
|---|---|---|
| lp_abs, hp_abs | Lp/Hp + 1.013(bar) | 密封式表压,口径 1 |
| te_sat, tc_sat | CoolProp R410A 饱和温度(lp_abs/hp_abs) | 口径 1 |
| sc_phys | tc_sat − Tl(**不减 1**) | 口径 2 |
| sh_phys | Ts − te_sat | 口径 3 |
| mode | AcState 映射(字典已锁);CompState=2 段打 special_program 标 | 状态码字典 |
| reversing | St(1 制热位 / 0 化霜位) | St 破案 |
| p_parasitic | PowerIn − PowerComp − 风机估算 | 口径 9 |
| comp_slip, fan_slip | 指令-反馈归一化偏差 | 口径:指令/反馈对 |
| tcs_gap | tc_sat − Tcs(目标追踪缺口) | 口径 10 |

时间戳统一 UTC 存储 + tz 列(时区口径未定前按样本 −05:00 摄取并打 tz_unverified 标)。

**C2 Zoho 标签契约**:fault_label_candidates schema(26 列)+ v2 六修复(明文列删除、日期钳制 [2018, 今日]、队列重排权重、series 过滤标、编码统一、匹配窗口字段)。产出表:`labels(sn_hash, fault_family, event_date, date_source, confidence, sn_status, review_state)`。

**C3 平台事件契约(桩)**:`events(sn_hash, ts_utc, fault_code, code_meaning, protection_bit, severity, cleared_ts, model, telemetry_first_date, telemetry_last_date, fw_version)`。字段映射层隔离实际 schema 差异。

**C4 实验室/产线契约**:结构与 C1 一致(已确认);实验室数据附加 `test_condition` 标签列(AHRI 工况点/极限工况类型),产线数据附加 `station, test_step`。到货后首个动作是 schema diff 脚本,任何列差异在此层吸收。

**C5 输出契约**:每条诊断 = `{sn_hash, 时间窗, 故障假设, 证据[特征,方向,幅度], 反证[], 置信度, 现场核查清单[], 版本号}`。出域物仅限:本契约的聚合统计、模型文件、评估报告。行级数据与 sn 明文永不出域。

## 4. 模块规格

每模块给输入/输出/算法要点/验收标准(DoD)。验收全部可在现有样本或桩数据上执行,不等外部交付。

**M-CONV 物理转换库**
输入 C1 原始列 → 输出派生列。算法:上表。
DoD:对 5 小时样本,sh_phys 与控制器上报 Sh 的差 ≤0.15 K(P95);sc_phys − 上报 Sc ≡ 1.0±0.1;化霜段(St=0)自动豁免一致性检查。

**M-SEG 状态分段与稳态检测器**
输入:状态列 + CompRps/Exv 时序。输出:每行 `segment_id, segment_type(run/off/defrost/special/transition), steady(bool)`。
算法:状态机按字典切段;真化霜判据 = CompState=2 ∧ St 翻转(Th 尖峰做校验不做判据);CompState=2 ∧ St=1 → special(剔除);稳态 = run 段内进入 >5 min ∧ CompRps 与 Exv 的 2 min 滚动标准差双低于阈值(阈值在样本上标定,实验室瞬态数据到货后重标)。
DoD:样本上 2 段真化霜、5 段 special、启停边界全捕获,零漏判;稳态覆盖率报告(样本预期 ~55–65%)。

**M-SENSE 传感器信任层**
输入:稳态段派生列(+产线指纹,到货前留桩)。输出:每 SN 每传感器 `trust_score, drift_flag`。
算法:压力-温度一致性(化霜稳定期 St=0 且盘管温度平台期,Th ≈ tc_sat 交叉验证);停机平衡检验(停机 >4h 后,冷媒侧各温度收敛 ∧ te_sat ≈ 最冷测点);Sh 基线自漂检测(逐机,月尺度);产线指纹对照(桩)。
DoD:注入式测试——对样本人工加 ±2K 偏置/斜坡漂移,检出率 100%,零误报。

**M-FEAT 特征库**
输入:稳态段。输出:同工况分箱特征表。分箱轴:`mode × Ta(2K 步)× CompRps(10 档)`,箱内聚合。
锁定特征清单:exv_resid(箱内 Exv 相对基线偏移)、sc_resid(sc_phys 剥偏置残差)、capacity_resid(Qh/Qc 相对基线)、approach(模式依赖:制冷 tc_sat−Ta,制热 Ta−te_sat)、th_coil_resid(TH 相对模式基线,制热=结霜/蒸发通道)、comp_slip、fan_slip、power_resid(PowerComp−PowerCompTheo)、p_parasitic、tcs_gap、i_resid(电流残差,**V1 入协变量**)、defrost_freq(化霜频率,结霜类前兆)。
DoD:样本全量特征无 NaN 泄漏;每特征附单元测试(方向性:人工注入充注减少的模拟扰动,exv_resid↑ sc_resid↓ 符号正确)。

**M-BASE 三层基线引擎**
L1 SKU 包络:XGBoost 回归,输入(Ta, CompRps, Exv, FanRpm, mode)→ 各状态量期望,实验室数据拟合,每 SKU 一套(共 6)。L2 出厂指纹:产线数据的部件级统计(传感器偏置、电气特征),**明确排除充注状态**。L3 现场基线:装机(首条遥测)后 30 天稳态分箱统计 + 滚动更新(变点保护:检出漂移期间冻结基线更新,防基线被故障污染)。
DoD:L1 在实验室留出工况点上 MAPE ≤5%(容量)/ ≤1K(温度类);L3 逻辑在样本上跑通(30 天窗降级为 5 小时演示)。

**M-DRIFT 漂移检测**
每(SN × 特征 × 箱)CUSUM + 鲁棒 Z,输出漂移起点、方向、幅度、斜率。学习/故障双通道判别:tcs_gap 与温控信号统计正常 ∧ 执行器漂移 = 学习侧,挂低优先;冷媒侧残差同向偏移 = 故障侧。
DoD:合成注入(斜坡 30/60/90 天三速率)检出延迟与误报率报告;阈值参数化,留待 M4 用真实标签标定。

**M-DIAG 诊断组合**
症状矩阵(初版按物理先验硬编码,冷媒不足 = exv↑ sc↓ capacity↓ lp↓;M4 后由树模型替换)→ C5 五件套输出。含室内侧非特异旗标逻辑(冷媒侧整体偏移但无单一故障签名匹配)。
DoD:桩标签上端到端产出合法 C5 记录。

**M-LABEL 标签连接引擎**
输入 C2 + C3。窗口:平台事件 ∈ [工单事件日 −21d, +3d];验证键 Error Code 抽检;闭环判定 = 维修动作日后 14 天内目标残差回归基线带。输出:`train_labels(sn_hash, family, onset_est, repair_date, closure_verified, tier)`。
DoD:桩数据全路径 + 边界用例(多工单同 SN、跨窗口、无恢复记录)测试通过。

**M-VALID 验证机架**
按设备切分的 leave-units-out;Yuill-Braun 五分类(正确/漏检/误报/误诊/无响应)+ 按故障影响加权;对照双基线:CoolantLackLimit 提前量、Title 24 目标过冷度法超额收益。
DoD:机架在合成数据上产出完整评估报告模板。

**M-ZOHO-V2**:六修复(v1.1 §5 清单)+ 重排队列导出。DoD:重跑 11,772 行,越界日期归零,重排后队列前 500 行中冷媒类占比 ≥30%。

## 5. 项目规划(双轨制)

**组织轨(全部 Alan 持有,工程无法替代)**:
- O1 实验室 + 产线数据调取——**与 D-1 解绑,立即启动**:公司内部测试数据,无客户 PII,不该走同一审批;它单独解锁 M2。
- O2 D-1 第一批次审批(冷媒 668 hash + 对照 ~1400 台)——关键路径起点,本周启动。
- O3 HMAC 盐协议(与平台方统一)——O2 的前置,盐不一致连接全败。
- O4 附录 A 转发固件团队(3 问)/ O5 附录 B 转发平台方 / O6 series→SKU 映射表 / O7 复核 1.5 人日排期(用 M-ZOHO-V2 重排后的队列,不用现队列)。

**工程轨里程碑**:

| 里程碑 | 内容 | 依赖 | DoD |
|---|---|---|---|
| M0(第 1–2 周) | M-CONV + M-SEG + M-ZOHO-V2 | 无 | 三模块样本验收通过 |
| M1(第 2–4 周) | M-FEAT + M-DRIFT + M-LABEL(桩)+ M-VALID 骨架 + M-SENSE(注入测试版) | 无 | 全模块合成/桩验收;端到端演示:样本进、C5 出 |
| M2 | M-BASE L1(6 SKU 包络)+ 稳态阈值重标 + L2 指纹 | **O1** | L1 留出点达标 |
| M3 | 摄取第一批次;L3 基线;M-SENSE 实跑;M-LABEL 实连 → **平台匹配率 + 每类最终落档**(B-3 收口) | O2+O3(+O6) | 匹配率报告;冷媒类落档判定 |
| M4 | 冷媒样板:标签标定 M-DRIFT 阈值 → 树模型 → leave-units-out → **CoolantLackLimit 提前量**(第一个价值数字) | M3 + O7 | 提前量报告 + 五分类评估 |
| M5 | FPR 实测(健康对照机全量跑)→ 分档放行决定;第二类别(风机电机)复制流程 | M4 | 放行评审材料 |

**关键路径声明**:M0–M1 零外部依赖,约 4 周;之后工程轨命脉全在 O2——**D-1 不批,M2 靠 O1 还能续命 2 周,M3 起全线停摆**。所以组织轨 O1/O2/O3 三件事的启动时间,比任何工程决定都更决定项目日历。

**协作模式(沿用已验证协议)**:模块代码由我逐个产出,你方工程师在隔离环境执行;行级数据不过境,回流物 = 验收结果与聚合报告;每里程碑收口时更新方案文档版本。

## 6. 风险登记(具体到环节与后果)

1. **盐不一致(O3 失守)**:M3 连接零匹配,症状是"匹配率 =0%"而非报错——摄取时先做 10 个已知 SN 的双边 hash 对拍,再跑全量。
2. **日期垃圾未修先连接**:195 张越界票把窗口匹配拖入 1997–2012 空档,平白拉低匹配率——M-ZOHO-V2 是 M-LABEL 实连的硬前置。
3. **实验室数据结构偏差**:C4 假设"结构一致"来自口头确认——到货第一步跑 schema diff,差异吸收在契约层,不许渗入模块。
4. **multi 未解歧即入训练**:668 上限里混错 SN 会给冷媒类注入错误正样本——M4 训练集只收 `review_state=confirmed` 或 `sn_status=valid`,multi 未解歧样本只进候选池不进训练。
5. **L3 基线被在途故障污染**:装机即带故障的机器,其"30 天基线"就是故障态——L3 建立期同时跑 M-SENSE 与保护位检查,异常机基线打 quarantine 标。
6. **样本外 SKU 渗入**(南方热泵/侧出风):O6 映射表未到前,Unknown 系列一律隔离池,不入 6-SKU 任何基线。
